# 用 grep 打败向量搜索？AI 圈被一篇论文整沉默了 😶

做 RAG 的朋友，先别划走。

你们天天卷的嵌入模型、向量库、重排序——一篇新论文告诉你：**一个 1973 年的老命令行工具，在多数场景下赢了你精心调优的语义搜索。**

这个工具叫 grep。

---

## 📊 实验结果长啥样

研究团队在 116 道长记忆问答题上做了测试，把 grep 和向量搜索放在一起比。

**当搜索结果直接塞进对话上下文时（inline 模式），grep 全面胜出，无一例外。**

几个数据感受一下：

- Chronos + Claude Opus 4.6：grep **93.1%** vs 向量 83.6%
- Codex CLI + GPT-5.4：grep **93.1%** vs 向量 75.9%
- Gemini CLI + Flash-Lite：grep **87.1%** vs 向量 67.2%

差距从 10 个点到 23 个点不等，不是小赢，是碾压。

---

## 🤯 比结果更炸裂的发现

论文里有个数据让我愣了一下：

同样是 Claude Opus 4.6 + grep，换了个 Agent 框架，准确率从 93.1% 掉到 76.7%。

**差了 16 个点。**

这比你从 grep 换成向量搜索造成的损失还大。

换句话说：**你用什么框架，比你用什么检索方式，影响更大。**

因为框架决定了系统提示词、工具描述、结果格式化——这些加在一起，才是"检索"真正发生的地方。

---

## 🔄 反转：换个模式向量搜索能反超

把搜索结果从"直接塞进对话"改成"写到文件让模型自己去读"，情况就变了。

最极端的例子：Codex + GPT-5.4

- inline grep：**93.1%**
- 写文件后 grep：55.2%（跌了快 38 个点！）
- 写文件后向量搜索：67.2%（反超了）

原因是"找到文件 → 打开 → 整合 → 回答"这条链路，对某些模型来说是个真实的能力考验。检索再好，模型读不懂也白搭。

---

## 🔊 加噪音测试：谁更抗压？

往数据里疯狂加干扰项，从 5 个加到 66 个。

有意思的规律：**噪音少的时候向量搜索更准，噪音多了 grep 反而更稳。**

因为语义搜索容易被"话题相近但内容无关"的东西带偏，而 grep 的精确匹配在大海捞针时反而更靠谱。

---

## 💡 为什么老工具能赢？

答案很简单：这类题的答案往往是"字面证据"——具体日期、人名、数字。

grep 直接命中，不需要经过嵌入模型那一层。

向量搜索的"语义理解"在这里反而成了负担：你问"2023年3月15日发生了什么"，不需要语义相似，你需要精确匹配。

> 🎯 grep 是追踪者——给线索就能找到
> 
> 🌐 向量搜索是联想者——网撒得大，干扰也多

---

## 🛠️ 对你意味着什么？

**做 RAG 的：** 别默认向量搜索是唯一答案。精确信息提取场景，先试试 grep，成本更低效果可能更好。

**选框架的：** 框架本身对效果的影响不比检索策略小。在真实 benchmark 上评估框架，比反复调嵌入模型更值。

**搭评测的：** 评测不能只报"BM25 vs ANN"，必须同时报告框架和结果返回方式，否则数字没有可复现性。
