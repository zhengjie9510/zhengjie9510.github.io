# 项目环境与工具偏好

## 命令执行优先原则

- **优先使用系统命令（Shell）** 完成文件操作、文本处理、环境检查等任务，避免不必要的脚本封装。

## Python 环境隔离原则

- **严禁在系统 Python 环境下安装任何依赖**。
- 所有 Python 包必须安装到 **当前项目目录下的虚拟环境**（`.venv/`）中。
- 执行 Python 脚本或安装依赖前，先确认已激活 `.venv` 虚拟环境；若未激活，使用 `.venv/bin/python` 或 `.venv/bin/pip` 的完整路径调用。

# 写作通用规范

适用于所有 Markdown 内容的编辑与创作。

## 内容组织

- **先定义，再展开**：读者应该在进入"为什么需要""如何实现"等技术细节之前，先明白"这到底是什么"。
- **最小改动**：已有内容只做必要的突出或概括化调整，不新增大段内容。
- **不提前剧透**：概念定义句应避免列举后文才会详细讲解的具体技术手段，用概括性表达替代。

## 语言与术语

- **引号**：中文正文中的引用、比喻、强调使用中文引号（“”‘’）；front matter、HTML 属性、代码块、英文术语保留英文引号。
- **术语统一**：正文中统一用中文表达，不混用英文缩写。
- **避免生僻外来借词**：优先使用大众日常熟悉的中文表达，不使用小众圈子或外语借词（如日语借词）替代日常词汇。
- **同义词不堆砌**：语义重复时选一个即可。
- **不做价值判断**：用"还有一种做法"而非"更合理的做法"。

## 排版

- **层级与表情**：中文数字编号（一、二、三）管层级，表情符号管趣味，不可替代编号。
- **表情适度**：小节标题或关键标记可加表情符号增加趣味性，不给正文每段都加。
- **标题风格**：章节标题作为结构导航，避免过度口语化、俚语化或网络梗式表达，优先简洁中性。

# 学习与记忆

从用户反馈中持续学习，并将学到的规则沉淀到项目文件中。

## 核心原则

- **学习是优先事项**：用户的每次纠正都是改进机会，不要只修表面问题，要理解背后的本质原因。
- **编码为模式**：把"这次怎么改的"提炼成"以后遇到类似情况该怎么做"的规则。
- **立即更新**：收到反馈后，优先把提炼后的规则写入 `AGENTS.md` 或对应的技能文件，再去做其他事情。
- **一次纠正，永久遵守**：如果用户指出过某个问题，后续工作中应主动避免，不需要等用户再次提醒。

## 什么应该记录

- 用户明确要求记住的信息（如偏好、习惯）。
- 用户描述的角色或行为方式（如"你要先分析再动手"）。
- 用户对工作成果的反馈（哪里不对、为什么不对、怎样才对）。
- 用户提供的项目上下文（如工具用法、工作流约定、编码风格）。
- 在任务中发现的新模式或偏好。

## 什么不应该记录

- 临时或瞬时的信息（如"我现在有事要出门"）。
- 一次性的任务请求（如"帮我查个数据"）。
- 简单问题或寒暄（如"今天星期几""谢谢"）。
- 任何凭证（API Key、密码、Token 等），绝不记录。
