智能体之所以有用，是因为它们能在现实世界中采取行动，帮我们实现工作自动化。但要让智能体可靠地执行有价值的工作，光有一个好模型还不够——它需要一套精心设计的**外部工作框架（Harness）**，而这套框架必须与具体的任务场景相匹配。

核心的智能体算法很简单：给 LLM 上下文，让它在循环中调用工具，直到任务完成。这是最基础的循环。

但它远不是驱动智能体的唯一循环。《Loopcraft: The Art of Stacking Loops》这篇文章提出：你可以不断**叠加和延伸循环**，从而构建出更强大的智能体。

下面就从最内层的智能体循环开始，逐层向外拆解。

## 第一层：智能体循环（Agent Loop）

在最核心的层面，智能体只做一件事：不断重复 **调用工具 → 观察结果 → 调整决策**，循环往复，直到任务完成。

```
[ 用户输入 ] ──> [ 思考/规划 ] ──> [ 调用工具 ] ──> [ 观察结果 ] ──> [ 是否完成？]
                     ^                                 │
                     └─────────────────────────────────┘
```

在这个最基础的循环里，**工具（Tools）** 是关键——没有工具，模型只能聊天；有了工具，它才能在现实世界里采取行动。

**【场景实例】** 你对智能体说：“帮我订一张明天下午从北京飞上海的机票。”

第一步，它调用航班查询 API，返回了 25 个航班。结果太多，它判断需要缩小范围。
第二步，调用筛选工具，限定“直飞 + 下午出发”，剩下 4 个。
第三步，4 个还是没法选，它逐一查询每个航班的历史准点率。
最后，综合价格、时间和准点率，给出推荐：“CZ1234，下午 2 点直飞，准点率 96%。”

每一步都在重复同一个模式：**看结果 → 判断 → 调工具 → 再看结果**。这就是智能体循环。

## 第二层：验证循环（Verification Loop）

基础的智能体循环确实能把工作做完，但由于大模型的随机性，它在第一次运行时并不总是能产出 100% 正确或一致的成果。在企业级应用中，当结果的确定性至关重要时，我们就必须在它的外层包裹一个"验证循环"。

验证循环的作用是：检查智能体的输出，如果发现没有达到预设标准，就将具体的反馈和修改意见发回给模型，让其重新生成。

```
                     ┌──────────────────────────────────┐
                     ▼                                  │ (不合格，附带反馈重试)
[ 循环 1: 智能体 ] ──> [ 输出结果 ] ──> [ 自动评分/验证器 ] ──> [ 最终合格输出 ]
```

验证循环中引入了"评分器（Grader）"的概念。评分器会根据既定的规则对智能体的产出进行打分。它可以是确定性规则的代码检测，也可以是智能体化的评估（即经典的"LLM 作为裁判/评审"模式）。

**【场景实例】** 在上述"文档智能体"中，评分器会在智能体每次尝试后自动运行测试，检查新生成的文档中：所有超链接是否都能正常打开？所有的持续集成（CI）检查是否通过？代码改动的范围是否严格限制在用户请求的范围内？通过这一层验证，系统无需人工干预就能自动拦截并修正大部分低级错误。

⚖️ **设计折中：** 引入验证循环必然会增加单次任务的响应延迟和 Token 成本。但当"质量"的优先级高于"速度"时（绝大多数商业生产场景都是如此），这个代价是非常值得的。

## 第三层：事件驱动循环（Event-Driven Loop）

智能体开发中，最关键的工程环节之一就是集成层（Integration Layer）。如何将智能体无缝融入现有的企业生态系统中，让它在后台默默工作，而不是像一个聊天框一样等你去唤醒？这就是第三层——事件驱动循环。它负责将智能体与业务系统、工作流连接起来。

```
[ 外部事件 (Webhook/定时任务/消息) ]
               │
               ▼
       [ 激活事件驱动循环 ] ──> [ 触发循环 1 & 2 运行 ] ──> [ 自动更新现实系统 ]
```

事件一旦被触发（例如：新文档上传、定时任务启动、系统收到 Webhook 通知、甚至 Slack/钉钉里有人 @ 了它），智能体就会自动启动。此时，智能体不再是用户必须手动调用的孤立工具，而是变成了一个在大型系统内部持续运转、主动服务的组件。

**【场景实例】** 在我们的日常工作中，文档智能体被挂载在沟通软件（如 Slack）的特定频道中。只要团队成员在这个频道内发送了一条消息，事件驱动机制就会立刻捕捉到，并在后台悄悄唤醒文档智能体进行处理、验证、并最终提交修改。

## 第四层：自优化循环（Hill Climbing Loop）—— 实现自我进化

前三个循环实现了"工作与执行的自动化"。而第四个循环，也是最具有变革意义的一层，实现了"改进与优化的自动化"！

智能体的每一次运行，都会在系统后台留下一条详细的运行追踪记录（Trace）：模型每一步说了什么、调用了什么工具、评分器给出了什么反馈。这些记录中蕴含着极高的优化价值。

自优化循环会运行一个专门的分析智能体去审阅这些运行记录，找出模型在哪一步容易犯错、哪部分提示词不够清晰、哪个工具调用效率低下，然后自动重写、优化基础智能体（第一层）的配置（包括微调提示词、工具定义或评分标准）。之所以叫 Hill Climbing（爬山），是因为它像爬山算法一样，每次都在现有基础上向更好的方向迭代一小步。

```
        ┌────────────────── 自动重写/优化配置 ─────────────────┐
        │                                                     ▼
[ 循环 1 / 2 / 3 运行 ] ──> [ 产生 Trace 记录 ] ──> [ 分析智能体 ]
```

**【场景实例】** 当我们让分析智能体去复盘上百次"文档智能体"的运行记录时，如果发现智能体多次因为同一个 Prompt 表达不清而导致评分器打回重写，分析智能体就会自动拟定一份优化后的 Prompt，并直接更新到文档智能体的底层配置中。每一次外层的自优化循环转动，都会让内层的智能体变得更加聪明、精准。

🚀 **未来的想象空间：** 调整 Prompt 和工具配置只是自优化循环的起点。对于使用开源模型的团队，自优化循环甚至可以将这些运行数据和评估结果作为训练信号，去进行 RL（强化学习）微调，从而直接升级大模型本身。

## 核心基石：人机协同（Human-in-the-Loop）

自动化并不意味着完全排除人类。在上述循环的每一个层面上，人类的专业知识和主观能动性依然有着无法替代的独特价值：

- **在智能体循环中：** 在触发敏感行动（如数据库删除、资金转账）之前，必须暂停并请求人类确认。
- **在验证循环中：** 对于主观性极强的产出，人类可以扮演那个"终极评分器"，判断表达的语气和调性是否合适。
- **在事件驱动循环中：** 在最终把结果交付给外部终端用户前，由人类进行一键审核。
- **在自优化循环中：** 分析智能体提出的 Prompt 改进或模型更新方案，在正式上线前应流经人类的 Review。

## 总结：四层循环堆叠图谱

| 循环层级 | 它在做什么 | 核心业务价值 |
|---------|-----------|-------------|
| 1. 智能体循环 | 模型根据任务，重复调用工具并观察结果直至结束 | 工作执行的自动化 |
| 2. 验证循环 | 对智能体输出进行标准打分，不合格则带反馈重试 | 确保产出质量与正确性 |
| 3. 事件驱动循环 | 通过业务事件、定时任务触发运行，融入企业系统 | 大规模系统级协同 |
| 4. 自优化循环 | 深度分析运行记录，自主优化系统配置与 Prompt | 系统的自我演进与优化 |

AI 领域的先行者们正在达成共识：智能体真正的天花板，不在于你用了多么强大的单一模型，而在于你围绕它们构建了怎样的"循环系统"。对于企业而言，越早构建起这种"人机协同、token 资本与人类判断共同产生复合效应"的学习循环，就越能在 AI 时代建立起难以被复制的竞争壁垒。
