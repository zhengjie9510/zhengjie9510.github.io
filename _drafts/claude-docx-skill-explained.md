## 📄 什么是「DOCX 技能」？

**「DOCX 技能」就是给 Claude 配备的一本「Word 文档操作手册」。** 无论是创建全新的商业报告、编辑现有的合同，还是处理复杂的格式要求，Claude 都会翻开这本手册，按照里面的规范来做事。

如果你还不清楚「技能」的概念，可以看看之前写的 [《别再复制粘贴提示词了！AI Agent Skill 到底是什么？》](/posts/ai-agent-skill/)，那篇文章有详细的介绍。

### 两种完全不同的工作方式

DOCX 技能涉及两种完全不同的操作模式：

- **从零创建** ——用代码写出一份格式完整的 Word 文档
- **编辑现有文件** ——把 .docx 拆开、修改内容、再重新打包

这两种方式在思路、工具、陷阱上都截然不同。下面逐一拆解。

---

## 🛠️ 从零创建文档：用代码「画」Word

当需要生成全新文档时，Claude 会用一个叫 **docx-js** 的 JavaScript 库，通过写代码的方式来「画」Word 文档。

### 这个过程是怎样的？

**把它想象成搭建乐高积木** ——每一段代码对应一个积木块：
- 这块是标题
- 那块是段落文本
- 另一块是表格或图像
- 最后一块是页脚

精确指定每个元素的位置、大小、颜色，然后一键生成 `.docx` 文件。

### 核心难点：DXA 单位系统

手册里反复强调的第一个难点是 **DXA 单位**（这是 Word 内部的长度单位）。

**最基础的规则很简单：**

> **1440 DXA = 1 英寸**

但问题来了——**页面大小、表格宽度、图像尺寸……全都要用 DXA 来指定。** 稍有不慎，文档就会排版混乱。

### DXA 的实际应用

比如常见的 US Letter 纸张：

- **宽度**：12,240 DXA（8.5 英寸）
- **高度**：15,840 DXA（11 英寸）  
- **内容区域宽度**（1 英寸边距）：9,360 DXA

手册里甚至专门列了一个表格，把常见纸张尺寸都换算好了。**这些都是真实踩过坑的教训。**

### A4 陷阱和横向排列陷阱

手册里强调得最多的**两个关键陷阱：**

**陷阱一：A4 默认值陷阱**

docx-js 库的默认纸张大小是 A4，但很多场景需要 US Letter。

如果你忘记显式指定纸张尺寸，最终生成的文档在美国用户那边就会排版错乱。**手册明确要求：必须显式设置页面大小，不能靠默认值。**

**陷阱二：横向排列（Landscape）的聪明做法**

普通人可能会想："我要横向排列，那就把宽度和高度对调"。但 docx-js 有个特殊设计：

> **你要传入纵向的尺寸（短边作为宽度，长边作为高度），然后设置 `PageOrientation.LANDSCAPE`，让库自己去对调。** 如果你自己对调了，反而会出错。

这种「反直觉」的设计说明了一个原则：**工具的便利性往往来自于设计者提前踩过的坑。**

### 表格的「双重宽度」规则

如果说 DXA 单位是第一个难点，**表格就是第二个巨大的坑。** 手册里用了这样的措辞：

> **表格需要「双重宽度」——既要设置表格宽度，也要设置每个单元格宽度，两者还要匹配。缺一不可。**

### 为什么这么复杂？

因为不同的平台（Word、Google Docs、在线编辑器）对表格的理解不一样。少数平台可能会忽略其中一个宽度设置，导致表格在某些地方显示错乱。所以必须两个都设置。

**还有一个细节值得注意：**

> **永远用 `WidthType.DXA`，绝对不要用百分比。** 因为百分比在 Google Docs 里会彻底失效。

这是跨平台兼容性的代价，看起来「冗余」的规则，其实是在追求用户体验的一致性。

### 列表陷阱：永远不要手动插入 Unicode 符号

**最容易犯的错误是直接手写：**

```javascript
// ❌ 错误做法
new Paragraph({ children: [new TextRun("• Item")] })
```

这样看起来是个列表，但其实是一个普通段落加个字符。一旦用户在 Word 里调整列表缩进或样式，这个「虚假的列表」就会显示混乱。

**正确做法是用 `LevelFormat.BULLET` 配合编号系统：**

```javascript
// ✅ 正确做法
numbering: {
  config: [
    { reference: "bullets",
      levels: [{ level: 0, format: LevelFormat.BULLET, text: "•", ... }] }
  ]
}
new Paragraph({ numbering: { reference: "bullets", level: 0 },
  children: [new TextRun("Real bullet item")] })
```

为什么手册里要这么强调？因为 AI 生成的列表"看起来对"但实际上是虚假的，这是最常见的隐患。

### 图像的「三件套」规则

插入图像看起来简单，但要求多：

```javascript
new ImageRun({
  type: "png",           // ⚠️ 必须指定
  data: fs.readFileSync("image.png"),
  transformation: { width: 200, height: 150 },
  altText: {             // ✅ 三个字段都要填
    title: "...",
    description: "...",
    name: "..."
  }
})
```

**最容易遗漏的是 `type` 参数。** 少了这一个参数，整个 ImageRun 就会创建失败。这看似是个小细节，但也是 AI 生成代码最常出错的地方。

## ✏️ 编辑现有文档：解包-修改-打包

当用户提供了现成的 Word 文档需要修改时，工作流就完全不同了。

### 核心思路：.docx 其实是 ZIP 压缩包

这是 DOCX 技能最聪明的地方。**一个 .docx 文件本质上是一个伪装的 ZIP 压缩包。** 里面装的是一堆 XML 文件，描述了文档的结构、内容、格式。

所以修改文档的思路就很清晰了：

1. **解包** ——把 .docx 当成 ZIP 解压，得到里面的 XML 文件
2. **编辑** ——直接修改 XML，改内容、改格式、加追踪更改
3. **打包** ——把修改后的 XML 重新压缩，生成新的 .docx

这个「看起来简单，实际很复杂」的三步流程，其实隐藏了大量的细节。

### 三步流程详解

**第一步：解包**

```bash
python scripts/office/unpack.py document.docx unpacked/
```

这一步做了四个重要的工作：

- ✅ 解压 ZIP 文件
- ✅ 把 XML 格式化，让它更易读
- ✅ **合并相邻的 run**（Word 里的最小格式单位），减少冗余
- ✅ **把智能引号转换成 XML 实体**（比如 `&#x201C;` 代表左引号），保证修改后不会变形

### 「合并相邻 run」的妙用

这里的「合并相邻 run」特别有意思。Word 在存储文本时，有时会把同一句话分成多个 run。解包时合并它们，编辑时就简单得多。

而且，把智能引号转换成 XML 实体，保证了**无论在什么版本的 Word、什么平台上，引号都能正确显示。** 这就是为什么手册后面会强调「用 XML 实体写智能引号」的原因。

**第二步：编辑 XML**

这是真正的「核心」——直接编辑 `unpacked/word/` 目录下的 XML 文件。

**手册里最强调的规则是：使用智能引号。** 

比如你要添加文本 `Here's a quote: "Hello"`，不能直接写这些字符，要用 XML 实体：

```xml
Here&#x2019;s a quote: &#x201C;Hello&#x201D;
```

### 为什么这么讲究？

因为普通的引号在某些场景下会被 Word 自动转换或者在复制粘贴时丢失。而 **XML 实体可以确保无论在什么版本的 Word、什么平台上，都显示出正确的排版效果。**

这体现了一个工程原则：**稳定性 > 简洁性。**

**第三步：打包和验证**

```bash
python scripts/office/pack.py unpacked/ output.docx --original document.docx
```

打包时做了三件事：

✅ 把 XML 文件重新压缩成 .docx  
✅ **自动校验** ——检查 XML 是否符合 OOXML 标准（Office 文档的官方标准）  
✅ **自动修复** 一些常见问题：比如某些 ID 超出范围、缺少空格保留标记等  

这个「自动修复」环节很关键。它能兜住很多「看起来正常」但实际不符合标准的 XML。这样最后生成的 .docx 文件就能在各种 Word 版本上正确打开。

### 追踪更改的复杂性

编辑现有文档时最难的是**追踪更改**（Tracked Changes）。

**简单情况：删除一段文本**

手册告诉你的做法是：

```xml
<w:del w:id="1" w:author="Claude" w:date="2025-01-01T00:00:00Z">
  <w:r><w:delText>deleted text</w:delText></w:r>
</w:del>
```

**复杂情况：删除整个段落**

这就复杂了。**不仅要标记文本内容被删除，还要标记段落符号本身被删除。** 否则接受更改后会留下一个空白段落。

为了避免这个问题，手册里要求：除了把内容放在 `<w:del>` 里，还要在 `<w:pPr><w:rPr>` 里加一个空的 `<w:del/>` 标签。这样才能保证段落彻底消失。

### 这说明了什么？

这种「双标记」的做法说明了一个深层的问题：**文档编辑的坑不是代码层面的，而是来自用户期望和实际行为的不匹配。**

## 🎨 内嵌的设计规范

和 PPTX 技能一样，DOCX 技能里也内嵌了一套**设计规范**。虽然没有列成一个清晰的「禁止清单」，但处处都体现了「好文档应该长什么样」。

### 字体选择：Arial 优先

手册建议用 **Arial** 作为默认字体。

为什么？因为 Arial 在所有系统、所有 Word 版本上都能保证正确显示，不会因为某台电脑没有安装字体而导致文档渲染错乱。

### 页眉页脚的两个陷阱

**陷阱一：不要用表格来做分隔线**

页脚要放两列内容（比如左边是日期，右边是页码），很多人的直觉是用表格。但**手册明确说不行**，因为表格单元格有最小高度，在页脚里会显示成空白的盒子。

正确做法是用 **Tab Stop**（制表位）——在同一行里用制表符隔开左右内容，然后设置右对齐的制表位。这样既简洁又兼容。

**陷阱二：页码要用特殊对象**

页码要用特殊的 `PageNumber.CURRENT` 对象，不能硬写数字。硬写 `1, 2, 3` 的话，页数多了就会全部显示成 `3`。

### 目录的 HeadingLevel 要求

生成目录（TOC）有个苛刻的要求：

> **只有用 `HeadingLevel` 标记的标题才会出现在目录里，自定义样式完全不行。**

为什么这么严格？因为 Word 的目录机制依赖于内置的标题级别系统，自定义样式虽然看起来像标题，但目录生成器认不出来。

**这说明了一个原则：当工具和用户期望有冲突时，规范会优先考虑兼容性而不是灵活性。**

## 💡 整体印象

研究完 DOCX 技能，有几个细节特别值得琢磨。

### 陷阱比规则更重要

**这个技能的 90% 的篇幅都在讲「不要这样做，要那样做」。** 比如：

- ❌ 不要用 Unicode 符号做列表
- ❌ 不要用百分比宽度的表格
- ❌ 不要在 Landscape 模式下手动对调宽高
- ❌ 不要在页脚里用表格

**为什么反复强调这些「不要」？** 因为它们都是 AI 最容易犯的错。用户看起来没问题的文档，实际上埋了雷，等到用户打开、编辑、转发的时候就会炸。

### 细致的单位换算

DXA、EMU（图像单位，914400 = 1 英寸）、pt（磅）……文档格式里有一大堆单位系统。

手册里不仅列了转换表，还列了常见页面尺寸的预算数值。**这说明什么？设计者不仅知道这些单位，还经历过换算错误导致的排版混乱，所以干脆把答案写进去。**

### XML 的「方言」特性

手册对 XML 结构有非常具体的要求。比如：
- `<w:pPr>` 里元素的顺序不能乱
- `<w:t>` 要自带 `xml:space="preserve"` 来保留空格
- RSIDs（修订 ID）要是 8 位十六进制数

这些都不是「理论上应该这样」，而是「Word 实际上要求这样」。它们来自对 OOXML 标准的深入理解，也来自对各种 Word 版本兼容性的反复验证。

### 跨平台的考量

表格宽度为什么要既设置表格宽度又设置单元格宽度？因为不同平台处理不一样。

正确做法看起来「冗余」，但其实是为了兼容所有平台的「最大公分母」。**这体现了一个工程原则：不是追求代码的优雅，而是追求最终用户体验的一致。**

Claude 能生成出一份专业的、可靠的文档，并不是因为它「天生懂文档」，而是因为**有人把这些规范、陷阱、最佳实践都编进了这个技能。**
