这两年，围绕 AI Agent 的新概念越来越多。

你会听到有人讲提示词工程，有人讲上下文工程，也有人讲 Harness Engineering。它们看起来都在讨论“怎么把大模型用好”，于是很多人学着学着就容易混乱：

- Agent 到底是不是一段会调用工具的提示词？
- 提示词工程和上下文工程到底差在哪？
- Harness Engineering 听起来像是个更大的新词，它和前两者又是什么关系？

如果不先把这些概念放回同一张图里看，后面越学越容易碎片化。

**这篇文章想先做一件更基础的事：不从零散技巧讲起，而是把 Agent、提示词工程、上下文工程、Harness Engineering 放到同一个系统里，一次性讲清它们各自负责什么。**

先给结论：

> 如果把大语言模型比作“大脑”，那么 Agent 不是这个大脑本身，而是一套围绕它搭起来的工作系统。
>
> 提示词工程，解决“人如何把指令下清楚”。
>
> 上下文工程，解决“这一轮到底该给它看什么”。
>
> Harness Engineering，解决“怎么让它稳定把事情做完”。

## 一、🤖 先定义：什么是 Agent

很多人第一次接触 Agent 时，容易把它理解成“更高级一点的聊天机器人”。

但如果说得更准确一点，**Agent 是一种能够感知环境、自主规划、进行推理、拥有记忆，并能够调用工具来完成特定任务的智能系统。**

不过，先有两个边界要分清：

- 它不是单独的模型本身。
- 它也不是一段一次性提示词。

它强调的不是“一次回答”，而是在一个持续运行的任务回路里不断往前做事。

一个最基本的 Agent，通常包含下面几样东西：

- 一个负责理解、推理和决策的核心能力单元。
- 一套给模型下达任务的规则，也就是提示词。
- 一部分当前要让模型看到的信息，也就是上下文。
- 一组可调用的工具，比如搜索、读文件、改文件、执行命令。
- 一套控制执行过程的外部安排，比如计划、校验、权限和审批。

所以，Agent 真正处理的，不只是“回答一个问题”，而是“围绕目标持续观察、判断、行动，再根据结果继续下一步”。

也正因为如此，**Agent 更像一个持续运转的工作回路，而不是一次性问答。**

它大致可以抽象成这样：

$$
\text{观察环境} \rightarrow \text{形成判断} \rightarrow \text{采取行动} \rightarrow \text{获得新反馈} \rightarrow \text{继续下一轮}
$$

任务一旦进入这种循环，问题就不再只是“模型够不够聪明”，而会立刻变成四个更实际的问题：

- 它有没有真正理解我的要求？
- 它这一轮到底看到了哪些信息？
- 它能调用哪些能力，边界在哪里？
- 它会不会中途跑偏、漏步，或者过早结束？

而这四类问题，恰好对应了后面我们要讲的几个概念。

## 二、🗣️ 提示词工程（Prompt Engineering）：解决“人如何把指令下清楚”

先从最熟悉的提示词工程开始。

**所谓提示词工程，本质上就是围绕“人如何把指令下清楚”所做的一系列设计。**

它处理的重点，不是“模型最后怎么说”，而是**人如何向模型表达要求**。也就是说，它解决的是模型开始行动之前的“理解问题”。

一个相对完整的提示词，通常会包含下面几个常见要素：

### 1. 角色设定

也就是先告诉模型，它此刻应以什么身份、什么工作方式来理解任务。

比如：

- 你是一名技术编辑。
- 你是一名代码审查助手。
- 你是一名擅长给初学者解释概念的老师。

角色设定不是为了“表演”，而是为了先把模型拉进一个更稳定的理解框架。

### 2. 背景说明

也就是把任务成立所必需的前提信息先交代清楚。

比如这篇文章面向谁、要发在哪里、已有内容是什么风格、哪些信息不能脱离原意乱改。

没有背景说明，模型虽然知道“要做什么”，却未必知道“是在什么前提下做”。

### 3. 任务描述

这是提示词里最核心的部分，也就是明确告诉模型“这次具体要完成什么”。

比如：

- 先通读文章，再提炼核心观点。
- 不要扩写没有依据的内容。
- 只做必要调整，不要整篇重写。

任务描述越具体，模型执行时就越不容易自行脑补。

### 4. 输出结构要求

也就是提前告诉模型，最后要按什么形式交付。

比如：

- 输出成标题加分点。
- 按“一、二、三”分节。
- 用表格输出。
- 返回 JSON。

很多时候，模型不是“不知道答案”，而是“没有按你想要的形式给答案”。这时问题往往不在能力，而在输出要求没有提前说清楚。

把这几个部分放在一起看，提示词工程并不神秘。它做的其实是一件很朴素的事：

**尽可能减少模型对任务的误解空间。**

比如同样是让模型“整理材料”，下面两种说法带来的结果就可能完全不同：

- 只说一句“帮我整理一下这份材料”。
- 明确说明“先通读，再提炼核心观点，最后改写成适合公众号发布的版本，不要扩写没有依据的内容”。

区别不在模型本身，而在你有没有把任务交代清楚。

所以，提示词工程通常会关心这些问题：

- 角色要不要先说清楚。
- 背景是不是交代充分了。
- 任务要不要拆开表达。
- 输出格式要不要提前约束。
- 哪些事能做，哪些事不能做。
- 出现不确定性时应该怎么处理。

除此之外，提示词工程里还有一些常见做法，也属于“怎么把指令下清楚”的范畴。比如，引导模型分步骤思考，这类做法常被归到 `Chain of Thought`；再比如，给模型一到两个示例，让它照着样例来，这类做法常被归到 `Few-shot`。它们本质上都在减少模型误解任务的概率，让输出更稳定。

不过，提示词工程虽然重要，但它解决不了全部问题。

因为就算你把任务说得再清楚，只要模型后面看到的信息不对，或者执行过程没有约束，它一样会跑偏。

这也是为什么，单靠“写好提示词”并不足以支撑一个真正可用的 Agent。

## 三、🧠 上下文工程（Context Engineering）：解决“这一轮该给它看什么”

如果说提示词工程处理的是“人如何下指令”，那么上下文工程处理的是另一个更容易被低估的问题：

**每一轮推理时，到底哪些信息应该被送进模型。**

这里最容易踩的坑，就是误以为“给得越多越好”。

但实际并不是这样。

如果给得太少，模型会失忆，不知道之前发生了什么；如果给得太多，模型又会被无关信息淹没，推理精度下降，甚至直接超过上下文窗口上限。

所以，**上下文工程的核心，不是机械地堆信息，而是持续决定“什么该进来，什么不该进来”。**

它通常包括这些做法：

- 对历史过程做总结和压缩。
- 把冗长工具结果替换成更短的表示。
- 把信息分成活跃记忆和外部存储。
- 只在需要时读取某一段文件或某一段记忆。
- 通过子智能体隔离海量中间过程。

到了这里，关注点已经不是“任务有没有说清楚”，而是“模型这一轮到底看到了什么材料”。

所以从系统视角看：

- 提示词工程更像是“交代任务的方法”。
- 上下文工程更像是“本轮推理的供料系统”。

前者决定理解方向，后者决定推理材料。

## 四、🧰 Harness Engineering：解决“怎么让它稳定把事情做完”

再往上一层，就是很多人最近才开始重视的概念：Harness Engineering。

**所谓 Harness Engineering，本质上就是人类围绕模型额外搭建的那套外部工作框架，用来控制它怎么执行、能执行到哪里，以及做到什么程度才算完成。**

如果说提示词工程是在“把指令交代清楚”，上下文工程是在“管理供料”，那么 Harness Engineering 处理的就是“如何把模型放进一套可控的执行流程里”。

它通常覆盖三类事情：

### 1. 规则层

也就是长期有效的行为要求。

比如：

- 先看清任务，再动手。
- 不清楚时先补充信息，不要乱猜。
- 做完之后先验证，再汇报。
- 高风险动作不能直接执行。

这类要求有时放在系统提示里，有时写在 `AGENTS.md` 这类规则文件里。

### 2. 工具与权限层

模型能不能读文件、改文件、联网、执行命令，不是模型自己决定的，而是系统是否给它这些能力。

更关键的是，**给工具不是重点，给边界才是重点。**

你让它能读哪里、改哪里、哪些动作需要确认，这些都属于 Harness 的范围。

### 3. 流程层

真正复杂的任务，往往不是败在“不会回答”，而是败在“不会稳定推进”。

所以系统通常还需要给它安排：

- 先列计划，再执行。
- 只推进当前最关键的一步。
- 每一步结束后检查结果。
- 关键节点交给人审批。
- 大任务拆给子智能体，主流程只接收结果。

看到这里，你应该能看出一个明显区别：

**提示词工程和上下文工程更多是在影响“这一轮怎么想”，Harness 则是在影响“整件事怎么做”。**

## 五、🧩 把四者放回一张图里看

如果把它们放回同一个系统，可以这样看：

### 1. Agent 是整体

它是那套真正运行起来的工作系统，而不是单独某一个部件。

### 2. 提示词工程决定“怎么交代任务”

它负责把角色、目标、格式和限制说清楚，让模型起步时别理解错。

### 3. 上下文工程决定“这一轮给模型喂什么”

它负责筛选、压缩、检索和加载信息，让模型每次看到的都是当前最有用的材料。

### 4. Harness Engineering 决定“整套系统如何约束执行”

它负责规则、工具、权限、计划、审批和校验，让模型不只是会答，还能稳稳地把事做完。

所以，用一句更完整的话说：

> Agent 是整套工作系统；提示词工程负责把指令下清楚；上下文工程负责把信息喂对；Harness Engineering 负责把执行过程管住。

## 六、📌 为什么真正难做的，往往不是提示词，而是系统协同

很多人刚开始做 Agent 时，容易把主要精力都放在提示词上。

这当然没错，只是它通常只是第一步。

因为一旦任务变成长流程，真正决定效果的，往往不再是某一句提示词有没有再润色一下，而是下面这些系统层面的问题：

- 模型有没有拿到正确的信息。
- 冗长信息有没有被提前过滤。
- 工具有没有边界。
- 执行过程中有没有计划和状态。
- 做完之后有没有验证闭环。

也正因为如此，很多 Agent 项目的问题，表面看起来像“模型不够强”，但更常见的原因其实是：

- 任务没交代清楚。
- 上下文喂得太乱。
- 外部工作框架没搭好。

换句话说，**提示词工程、上下文工程、Harness Engineering 这三者，不是谁比谁更高级，而是在补同一个系统里的不同短板。**

## 七、✅ 最后用一句话收尾

如果你把大语言模型看成“大脑”，那么：

- 提示词工程（Prompt Engineering）是教你怎么把指令下清楚。
- 上下文工程（Context Engineering）是教你怎么把材料给对。
- Harness Engineering 是教你怎么把流程管住。
- Agent 则是把这一切真正组装起来、让它开始工作的那套系统。

这也是为什么，理解 Agent，不能只盯着提示词；理解智能体工程，也不能只盯着模型本身。

真正重要的，是把“语言、信息、工具、流程”放回同一个系统里去看。
