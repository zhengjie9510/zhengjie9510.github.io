前几天我用 Claude 做了一个演示文稿，沟通好需求之后，它直接写了一段代码，没多久就输出了一份排版挺不错的 PPT。当时我就好奇：这背后到底是怎么运作的？

带着这个好奇心，我翻开了它用的"PPTX 技能"的内部文件，仔细研究了一番。今天把这份解读分享给大家。

---

## 什么是"技能"？

先说背景。Claude 的 Cowork 模式支持安装"技能"（Skill），你可以把它理解成给 Claude 配备的一本"专项操作手册"——里面写好了遇到特定任务时该怎么思考、用什么工具、遵守哪些规矩。

PPTX 技能就是专门处理 PPT 的那本手册。只要你跟 Claude 提到演示文稿、幻灯片之类的需求，它就会先翻开这本手册，按上面的流程来做事。

---

## 技能的整体目录结构

打开这个技能的文件夹，结构是这样的：

```
pptx/
├── SKILL.md            ← 主入口，Claude 第一个读的文件
├── editing.md          ← 基于已有模板修改 PPT 的工作流
├── pptxgenjs.md        ← 从零创建 PPT 的操作手册
└── scripts/
    ├── thumbnail.py    ← 把 PPT 生成缩略图，用于快速预览模板
    ├── add_slide.py    ← 复制或新建幻灯片
    ├── clean.py        ← 清理临时文件
    └── office/
        ├── unpack.py   ← 把 .pptx 文件"拆开"，方便修改
        ├── pack.py     ← 修改完之后重新"打包"成 .pptx
        ├── soffice.py  ← 负责把 PPT 渲染成图片
        └── validate.py ← 打包前做一次文件合法性检查
```

看起来文件不少，但核心逻辑就三条线：**主入口 → 两种工作流 → 一套幕后工具**。下面逐一拆解。

---

## SKILL.md：Claude 的"第一眼"

每次 Claude 要处理 PPT 需求，它读的第一个文件就是 `SKILL.md`。这个文件起"总控"作用，里面有三块关键内容。

**第一块：路由判断**

手册里有一张简单的对照表，告诉 Claude 根据情况走哪条路：

- 用户给了模板 → 去读 `editing.md`，按模板修改
- 从零开始 → 去读 `pptxgenjs.md`，用代码生成

**第二块：设计规范**

这是让我最意外的部分——手册里内嵌了一份相当完整的"PPT 视觉设计指南"。

配色方面，手册里列了 10 套提前搭配好的主题色盘，每套都包含主色、辅色和强调色。这样 Claude 不用自己随便选颜色，而是从经过设计的方案里挑一套最符合主题的。

排版方面，手册规定了标题要用多大字号、正文用多大、推荐哪几种字体组合、段落之间留多少空白，相当具体。

更有意思的是手册里的"禁止清单"——专门列出 AI 做 PPT 最容易犯的毛病，要求 Claude 主动避开：

- 不许每张幻灯片都用同一个布局（千篇一律，看起来无聊）
- 不许在标题下面加装饰横线（这是 AI 生成 PPT 的典型标志，一眼就看出来）
- 不许用米黄色背景（太默认了，没有设计感）
- 不许做纯文字幻灯片（每张都必须有图、图表或图标）
- 不许把正文居中对齐（居中只适合标题，正文要靠左）

这份清单说明了一个道理：AI 工具做得好不好，很大程度上取决于有没有人提前把这些"行业经验"写进去。

**第三块：质检流程**

手册规定，初版做完之后必须过一遍视觉检查——把每张幻灯片渲染成图片，逐一排查溢出、遮挡、对齐等问题。这个环节是强制的，不是可选项。

---

## pptxgenjs.md：从零造 PPT 的核心手册

当没有模板时，Claude 使用一个叫 PptxGenJS 的工具，通过写代码的方式来"画"PPT。`pptxgenjs.md` 就是这个工具的使用说明，也是 Claude 写代码时的直接参考文档。

你可以把这个过程想象成用乐高积木搭东西：每一段代码对应一个积木块——这块是文字框、那块是图表、另一块是背景色——精确指定每个元素放在哪个位置、多大尺寸、什么颜色。所有积木按坐标拼好之后，一键生成 `.pptx` 文件。

手册里有一节让我印象很深，叫"常见陷阱"——专门列出容易踩的坑，每条都给出了正确和错误的对比。这类内容其实比正常用法更有价值，因为它记录的是真实踩过的教训。比如颜色值如果多加了一个 `#` 符号，整个文件就会损坏；某个配置如果重复使用而不是每次重新创建，第二次调用时数据就会出错。这些问题光靠猜是猜不到的，必须有人提前写进来。

图表的处理原则也值得一提。手册里明确规定：只要 PPT 原生支持的图表类型（条形图、折线图、饼图等），就必须用原生图表，不允许截图代替——原因是用户拿到文件之后还能继续修改数据，截图就彻底死了。

---

## editing.md：基于模板修改 PPT 的流水线

当用户提供了现成的 PPT 模板，工作方式就不一样了。

核心思路是：`.pptx` 文件本质上是一个压缩包，里面装的是一堆描述"这张幻灯片长什么样"的文本文件。Claude 会把这个压缩包解开，直接修改里面的内容，再重新压缩回去。

完整流程分七步：

```
1. 生成模板缩略图，看清楚每个版式长什么样
2. 规划内容与版式的对应关系
3. 把 .pptx 解包，得到可以直接编辑的文件
4. 完成所有结构性操作（删幻灯片、复制幻灯片、调顺序）
5. 逐张修改每张幻灯片的内容
6. 清理解包过程产生的临时文件
7. 重新打包，生成最终 .pptx
```

这里有个有意思的细节：第五步"逐张修改内容"被特别标注为"如果有条件，多个幻灯片可以同时并行处理"。这是因为每张幻灯片是独立的文件，互不干扰，所以完全可以多个任务同时进行，提高效率。

## scripts/ 目录：一套幕后工具

### unpack.py 和 pack.py

这两个脚本是一对，负责 PPT 文件的"拆开"和"合上"。

`.pptx` 文件就是一个伪装成演示文稿的压缩包，解开之后里面是一堆描述幻灯片结构的文本文件。unpack.py 负责解压并整理格式，让内容更容易阅读和编辑；pack.py 负责改完之后重新压缩，并在打包前做一次结构检查，确保文件是合法的。

### thumbnail.py

这个脚本的工作是把 PPT 的每张幻灯片都渲染成小图，然后拼成一张网格图。用途是让 Claude（和用户）快速看到所有版式的样子，在正式修改之前先选好要用哪些布局。

### soffice.py

这是对 LibreOffice（一个免费的办公软件）的封装。它的核心工作是在没有界面的情况下，把 PPT 文件自动转换成图片，用于后续的视觉检查。你可以把它理解成一台"自动截图机器"。

### validate.py 和 schemas/ 目录

schemas 目录里存放的是 Office 文件格式的官方规范文档（由国际标准化组织发布）。validate.py 在打包时用这些规范来校验生成的文件是否合法，就像检查一份合同有没有漏签名、漏填章一样——确保最终输出的 .pptx 文件能被 PowerPoint 正常打开。

## 整体运行逻辑

把前面所有内容串起来，大概是这样一张图：

```
用户说出需求
      │
      ▼
SKILL.md（判断走哪条路）
      │
      ├── 有模板 ──→ editing.md
      │                  │
      │                  ▼
      │           解包 → 修改内容 → 清理 → 打包
      │
      └── 无模板 ──→ pptxgenjs.md
                         │
                         ▼
                    写代码 → 运行 → 输出 .pptx

两条路最终都要经过：
      渲染成图片 → 视觉检查 → 发现问题 → 修复 → 再检查
```

## 一点感想

研究完这个技能的内部文件，有几点印象比较深。

**技能不只是"会操作"，还内嵌了审美和品控。** 配色方案、字号规范、禁止清单——这些内容让我意识到，Claude 能做出还不错的 PPT，不是因为它"天生会设计"，而是因为有人提前把大量设计经验编码进去了。

**"踩坑记录"的价值不亚于使用说明。** pptxgenjs.md 里那些"常见陷阱"的篇幅和正常用法差不多，每条都有正确和错误的对比。这是一种降低出错率的设计方式，也是工具能稳定运行的重要原因之一。

**每个工具各司其职，没有重叠。** PptxGenJS 负责从零创建，LibreOffice 负责渲染和格式转换，Python 脚本负责文件的解包、打包和验证——分工清晰，互不干扰。

下次再让 Claude 做 PPT，应该会更清楚它在"想什么"了。
