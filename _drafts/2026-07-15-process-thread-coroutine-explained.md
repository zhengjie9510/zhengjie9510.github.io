今天不背书。一个**煎饼摊**，加几段**能直接跑的 Python**，让你彻底搞懂。

## 一、进程：一个煎饼摊，就是一个进程

你在小区门口摆了个煎饼摊：一辆三轮车、一个鏊子、一桶面糊、一筐鸡蛋，还有你自己。

**这一整套家当加上你，就是一个"进程"。**

生意好，你去隔壁小区又开了一个摊——**这就是"多进程"**。

两个摊完全独立：隔壁鸡蛋用完了不影响你，隔壁鏊子坏了你照摊，隔壁被城管掀了你屁事没有。

**但代价是：贵。** 开第二个摊得再买一整套设备。而且两个摊沟通很麻烦，只能打电话（这在计算机里叫 IPC，进程间通信）。

### 代码：两个进程，各自有各自的"鸡蛋筐"

```python
from multiprocessing import Process
import os

eggs = 100  # 鸡蛋筐

def work(name):
    global eggs
    eggs -= 50           # 我在我自己的摊里，用掉 50 个鸡蛋
    print(f"{name} 摊(PID={os.getpid()}) 剩余鸡蛋: {eggs}")

if __name__ == "__main__":
    p1 = Process(target=work, args=("A",))
    p2 = Process(target=work, args=("B",))
    p1.start(); p2.start()
    p1.join();  p2.join()
    print(f"主进程(PID={os.getpid()}) 剩余鸡蛋: {eggs}")
```

**输出：**

```
A 摊(PID=1002) 剩余鸡蛋: 50
B 摊(PID=1003) 剩余鸡蛋: 50      <-- 注意！不是 0，是 50
主进程(PID=1001) 剩余鸡蛋: 100    <-- 主进程的鸡蛋一个没少
```

看到了吗？**两个进程各减 50，结果都是 50，主进程还是 100。**

因为它们**根本不是同一筐鸡蛋**。每个进程都把鸡蛋筐复制了一份，各玩各的。

> 📌 **进程 = 一整套独立的家当。安全、隔离，但重、贵、不好沟通。**

## 二、线程：一个摊上雇几个人，就是多线程

开分摊太贵了。换个思路：**摊子还是这一个，但我多雇几个人。**

- 小王摊面糊
- 小李打鸡蛋
- 小张收钱打包

**这三个人，就是三个"线程"。**

好处立刻就来了：**便宜**（不用再买鏊子），**沟通方便**（喊一嗓子就行）。

**因为他们用的是同一桶面糊、同一筐鸡蛋、同一个鏊子——这就是"线程共享内存"。**

### 代码：三个人共用一筐鸡蛋

```python
from threading import Thread

eggs = 100  # 同一筐鸡蛋

def work(name):
    global eggs
    eggs -= 30
    print(f"{name} 拿完了，剩余鸡蛋: {eggs}")

ts = [Thread(target=work, args=(n,)) for n in ["小王", "小李", "小张"]]
for t in ts: t.start()
for t in ts: t.join()
print(f"最终剩余: {eggs}")
```

**输出：**

```
小王 拿完了，剩余鸡蛋: 70
小李 拿完了，剩余鸡蛋: 40
小张 拿完了，剩余鸡蛋: 10
最终剩余: 10       <-- 真的少了 90 个！
```

**跟进程完全不一样了。** 三个人是真的在**同一筐鸡蛋**里拿，拿一个少一个。

这就是线程的好处：**共享数据，不用打电话沟通。**

**但——问题也来了。**

### 问题一：抢东西（竞态条件）

筐里只剩最后一个鸡蛋。小李伸手去拿，小张也伸手去拿。两人**同时**看到"还有一个"，都以为自己能拿到——结果打起来了。

**这就是竞态条件（Race Condition）。** 用代码复现一下：

```python
from threading import Thread

count = 0

def add():
    global count
    for _ in range(1_000_000):
        count += 1     # 看着是一步，其实是"读 → 加 → 写"三步！

ts = [Thread(target=add) for _ in range(2)]
for t in ts: t.start()
for t in ts: t.join()

print(count)   # 你以为是 2000000
```

**实际输出（每次都不一样）：**

```
1247831
```

**为什么？**

因为 `count += 1` 不是一个动作，是**三个**：

```
1. 读出 count 的值   (小李看到还剩 5 个)
2. 加 1
3. 写回 count        (小李写：现在剩 6 个)
```

小李刚做完第 1 步（读到 5），**还没来得及写回去**，操作系统把他叫停了，换小张上。小张也读到 5，也算成 6，也写回 6。

**两个人各加了一次，结果只加了一次。** 数据就这么丢了。

### 怎么解决？加锁

给鸡蛋筐上把锁，谁拿谁先拿钥匙：

```python
from threading import Thread, Lock

count = 0
lock = Lock()      # 鸡蛋筐的锁

def add():
    global count
    for _ in range(1_000_000):
        with lock:         # 拿钥匙 → 开锁 → 拿鸡蛋 → 还钥匙
            count += 1

ts = [Thread(target=add) for _ in range(2)]
for t in ts: t.start()
for t in ts: t.join()

print(count)   # 2000000  ✅ 这次对了
```

**代价是：慢。** 因为所有人必须排队等钥匙。

### 问题二：死锁

更惨的：

- 小李攥着**鸡蛋筐的钥匙**，等**铲子**
- 小张攥着**铲子**，等**鸡蛋筐的钥匙**

俩人互相瞪着，谁也不松手。**摊子彻底停摆——这就是死锁。**

### 问题三：一个人出事，全摊完蛋

小王把整桶面糊打翻了——**不光是他，整个摊都干不了活了。**

**这就是线程的致命伤：一个线程崩溃，整个进程一起挂。**

（对比：两个独立的摊，一个打翻面糊，另一个照卖。这就是进程的隔离优势。）

> 📌 **线程 = 一个摊上的多个伙计。快、省、好沟通，但会抢东西、会死锁、一个出事全体完蛋。**

## 三、协程：一个人，但特别会"见缝插针"

雇人也有成本，人一多还打架（加锁、死锁），管理起来很烦。

于是你冷静观察了一下自己的流程：

```
1. 倒面糊摊开      —— 需要我，10秒
2. 【等面糊凝固】  —— 不需要我，30秒，我在干瞪眼  ← ❗
3. 打鸡蛋抹酱      —— 需要我，15秒
4. 【等煎饼定型】  —— 不需要我，20秒，我又在干瞪眼 ← ❗
5. 铲起来收钱      —— 需要我，10秒
```

**你发现一个惊人的事实：你有一大半时间在干等。**

那能不能——**在等第一个煎饼凝固的那 30 秒里，去倒第二个煎饼的面糊？**

当然能。

**恭喜，你刚刚发明了协程。你还是一个人，但产能翻了三倍。**

### 代码：先看"笨办法"（同步）

```python
import time

def make_pancake(name):
    print(f"{name}: 倒面糊")
    time.sleep(2)                    # 傻等 2 秒，人就杵在那儿
    print(f"{name}: 出锅！")

start = time.time()
for n in ["A", "B", "C"]:
    make_pancake(n)
print(f"总耗时: {time.time() - start:.1f} 秒")
```

**输出：**

```
A: 倒面糊
A: 出锅！
B: 倒面糊
B: 出锅！
C: 倒面糊
C: 出锅！
总耗时: 6.0 秒     <-- 一个一个来，2+2+2
```

**你在那儿干等了 6 秒，其中至少 4 秒纯属浪费。**

### 代码：协程版（见缝插针）

```python
import asyncio, time

async def make_pancake(name):
    print(f"{name}: 倒面糊")
    await asyncio.sleep(2)           # ⭐ "这2秒不用我，你先忙别的"
    print(f"{name}: 出锅！")

async def main():
    await asyncio.gather(
        make_pancake("A"),
        make_pancake("B"),
        make_pancake("C"),
    )

start = time.time()
asyncio.run(main())
print(f"总耗时: {time.time() - start:.1f} 秒")
```

**输出：**

```
A: 倒面糊
B: 倒面糊         <-- 没等 A 好，直接去倒 B 了！
C: 倒面糊
A: 出锅！
B: 出锅！
C: 出锅！
总耗时: 2.0 秒     <-- 从 6 秒变成 2 秒
```

**从头到尾只有一个线程、一个人。** 但因为在等待的时候跑去干别的了，时间直接从 6 秒压到 2 秒。

**关键就是那个 `await`。**

翻译成人话，`await` 就是你说的那句：

> **"这事儿得等，我先去忙别的了，好了叫我。"**

## 四、协程为什么不会打架？

回想线程的问题：抢鸡蛋、死锁。

**协程为什么没这些问题？因为从头到尾只有你一个人。**

- 鸡蛋不会被抢——只有你一只手在拿
- 不会死锁——不存在"另一个人"跟你抢
- **你自己决定什么时候切换**

这是协程和线程最本质的区别：

| | 线程 | 协程 |
|---|---|---|
| 谁决定切换 | **老板（操作系统）** | **你自己（代码里的 `await`）** |
| 什么时候切 | 随时，你干到一半也可能被叫停 | 只在你**主动**让出的时候 |
| 会打架吗 | **会**，要加锁 | **不会** |

**线程是抢占式的**：老板拿着秒表，时间一到就把你叫停。你可能刚把鸡蛋磕开一半就被打断了——这就是竞态条件的根源。

**协程是协作式的**：你自己心里有数，"我这会儿要等 30 秒，先去忙别的"。**你主动让出，主动回来。**

## 五、协程的死穴（划重点，这里最多人踩坑）

看到这你可能觉得：协程这么牛，以后全用协程算了？

**打住。**

假设有个顾客要求：**"给我现磨一斤芝麻酱。"**

你抱起石磨，吭哧吭哧地磨。

磨芝麻这活儿——**它不需要"等"，它需要你一直用力。**

你磨的这 10 分钟里：没法倒面糊，没法打鸡蛋，**整个摊子彻底卡死**，后面排队的全在干等。

### 代码：协程遇到"磨芝麻"

```python
import asyncio, time

async def grind_sesame(name):
    print(f"{name}: 开始磨芝麻")
    s = time.time()
    while time.time() - s < 2:       # ❌ 死循环在"算"，一直占着 CPU
        pass                          #    从头到尾没有 await，谁也插不进来
    print(f"{name}: 磨完了")

async def main():
    await asyncio.gather(grind_sesame("A"), grind_sesame("B"), grind_sesame("C"))

start = time.time()
asyncio.run(main())
print(f"总耗时: {time.time() - start:.1f} 秒")
```

**输出：**

```
A: 开始磨芝麻
A: 磨完了
B: 开始磨芝麻      <-- 只能等 A 磨完，B 才能开始
B: 磨完了
C: 开始磨芝麻
C: 磨完了
总耗时: 6.0 秒     ❌ 协程完全没用！又变回 6 秒了
```

**一模一样的 asyncio 代码，煎饼场景 2 秒，磨芝麻场景 6 秒。**

**为什么？因为磨芝麻的过程中，没有任何一个 `await` 让出的机会。**

> ## ⚠️ 协程只能解决"等"的问题，解决不了"干"的问题。

- **IO 密集**（有大量等待：等网络、等数据库、等文件）→ **协程神器**
- **CPU 密集**（一直在算：图像处理、加密、大数据计算）→ **协程完全没用**

磨芝麻这种活儿，你**必须**再雇个人（多线程），或者**必须**再开个摊（多进程）。

**一个人再怎么见缝插针，也变不出第二双手。**

## 六、新手第一大坑：在协程里写了阻塞代码

这个坑，值得单独拎出来说，因为**几乎所有人第一次写 asyncio 都会踩**。

```python
import asyncio, time

async def make_pancake(name):
    print(f"{name}: 倒面糊")
    time.sleep(2)         # ❌❌❌ 注意！是 time.sleep，不是 await asyncio.sleep
    print(f"{name}: 出锅！")

async def main():
    await asyncio.gather(make_pancake("A"), make_pancake("B"), make_pancake("C"))

start = time.time()
asyncio.run(main())
print(f"总耗时: {time.time() - start:.1f} 秒")
```

**输出：**

```
A: 倒面糊
A: 出锅！
B: 倒面糊
B: 出锅！
C: 倒面糊
C: 出锅！
总耗时: 6.0 秒     ❌ async/await 全白写了
```

**就差一个字，结果天壤之别：**

| 写法 | 含义 | 结果 |
|---|---|---|
| `await asyncio.sleep(2)` | "我要等 2 秒，**你先忙别的**" | ✅ 2 秒 |
| `time.sleep(2)` | "我要**坐下来睡** 2 秒" | ❌ 6 秒，全摊陪你睡 |

**在协程里，任何一个"同步阻塞"的调用，都会把整个事件循环卡死。**

所以用了 asyncio，就**必须**全套换成异步的：

- ❌ `requests` → ✅ `aiohttp` / `httpx`
- ❌ `time.sleep` → ✅ `asyncio.sleep`
- ❌ 同步的数据库驱动 → ✅ `asyncpg` / `aiomysql`

**混进去一个同步的，前功尽弃。**

## 七、所以到底怎么选？就问自己一句话

> # 你的程序，是在"算"，还是在"等"？

### 在"等" → 协程

爬虫在等网页返回。API 在等数据库。聊天服务器在等用户发消息。

**CPU 大部分时间在干瞪眼。协程能把这些等待的缝隙全榨干。**

一个线程用协程，能同时"照看"上万个煎饼——因为它们大部分时间都在鏊子上自己凝固呢。

### 在"算" → 多进程

图像处理、视频转码、模型训练、大规模计算。

**CPU 一直满载，一秒都没闲着。** 你唯一的出路是**加人、加摊**（利用多核）。

> ⚠️ **Python 用户特别注意：** 因为 **GIL** 的存在，Python 的**多线程在 CPU 密集任务上基本等于没用**（一个摊上雇了 3 个人，但老板规定同一时刻只准一个人动手）。CPU 密集任务在 Python 里**必须**用 `multiprocessing`。

### 又"算"又"等" → 混着来

**多开几个摊，每个摊里的人都会见缝插针。**

这就是现在所有高性能服务的标准架构：**多进程 + 协程**。

Go 的 goroutine、Nginx 的多 worker、Node.js 的 cluster——全是这个套路。

## 八、三个最常见的误解

**❌ 误解 1："协程比线程快"**

不对。**单个煎饼的制作时间一点没变。**

协程快的是"**切换**"——你从 A 煎饼转身去看 B 煎饼，几乎不花时间。而线程切换要惊动老板（操作系统），来回打报告，慢得多。

**协程提高的是吞吐量，不是单任务速度。**

**❌ 误解 2："用了协程就能跑满 8 核"**

不对。**你就一个人，长了 8 只手吗？**

单线程的协程，永远只在**一个核**上跑。想吃满多核，要么多进程，要么用 Go 那种自带"多线程 + 协程"调度器的语言。

**❌ 误解 3："协程里可以随便写代码"**

看第六节。`time.sleep` 那个坑，你迟早会踩一次。

## 九、存下这张表，比背八股文管用

| | 进程 | 线程 | 协程 |
|---|---|---|---|
| **比喻** | 多开几个摊 | 一个摊多雇人 | 一个人见缝插针 |
| **Python** | `multiprocessing` | `threading` | `asyncio` |
| **谁调度** | 操作系统 | 操作系统 | **你自己（`await`）** |
| **切换成本** | 最贵 | 中等 | **几乎免费** |
| **能开多少** | 几十个 | 几千个 | **几十万个** |
| **会打架吗** | 不会（各摊独立） | **会**，要加锁 | 不会（就一个人） |
| **一个崩了** | 别的没事 | **全完蛋** | **全完蛋** |
| **能用多核** | ✅ | ✅（Python 因 GIL ❌） | ❌ |
| **适合** | CPU 密集 / 要隔离 | CPU 密集（非 Python） | **IO 密集** |

最后念一遍，你就出师了：

> ## 🥞 进程管隔离，线程管并行，协程管并发。
> ## 🥞 在"算"就加人加摊，在"等"就见缝插针。
