# 项目环境与工具偏好

## 命令执行优先原则

- **优先使用系统命令（Shell）** 完成文件操作、文本处理、环境检查等任务，避免不必要的脚本封装。

# 写作通用规范

适用于所有 Markdown 内容的编辑与创作。

## 语言与术语

- **引号**：中文正文中的引用、比喻、强调使用中文引号（""''）；front matter、HTML 属性、代码块、英文术语保留英文引号。
- **术语统一**：正文中统一用中文表达，不混用英文缩写。
- **避免生僻外来借词**：优先使用大众日常熟悉的中文表达，不使用小众圈子或外语借词（如日语借词）替代日常词汇。
- **同义词不堆砌**：语义重复时选一个即可。

# 学习与记忆

从用户反馈中持续学习，并将学到的规则沉淀到项目文件中。

## 核心原则

- **学习是优先事项**：用户的每次纠正都是改进机会，不要只修表面问题，要理解背后的本质原因。
- **编码为模式**：把"这次怎么改的"提炼成"以后遇到类似情况该怎么做"的规则。
- **先判断是否值得沉淀**：收到反馈后，先区分它是长期稳定的规则，还是当前任务中的一次性取舍；只有前者才写入 `CLAUDE.md` 或对应技能文件。
- **一次纠正，永久遵守**：如果用户指出过某个问题，后续工作中应主动避免，不需要等用户再次提醒。
- **只记录项目级规则**：`CLAUDE.md` 只存放跨任务、跨文件、可复用的工作方式、约束和偏好，不记录某一篇文章、某一轮修改里的局部处理决定。

## 什么应该记录

- 用户明确要求长期记住的信息（如稳定偏好、固定习惯）。
- 用户描述的长期工作方式（如"先分析再动手"）。
- 跨任务可复用的项目约定（如工具用法、工作流约定、编码风格）。
- 多次出现、且已经证明是稳定偏好的模式。

## 什么不应该记录

- 临时或瞬时的信息（如"我现在有事要出门"）。
- 一次性的任务请求（如"帮我查个数据"）。
- 某一篇文章、某一个页面、某一次改稿里的局部取舍。
- 还没有证明会长期复用的单次反馈。
- 简单问题或寒暄（如"今天星期几""谢谢"）。
- 任何凭证（API Key、密码、Token 等），绝不记录。
