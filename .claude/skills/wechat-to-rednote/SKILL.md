# 公众号 → 小红书 改写技能

将微信公众号 Markdown 文章改写为适合小红书发布的版本，最终输出一个可直接导入 MD2Card 等工具的 `.md` 文件。

---

## 改写流程

### 第一步：理解原文

阅读用户提供的公众号文章，识别：
- 核心观点和结构
- 技术术语或专业表达（需要简化）
- 代码块、命令行等内容（需要去除或转化为口语描述）
- 表格（保留，但超过 4 列时考虑拆分）

### 第二步：内容改写

按以下规则改写：

**语气与风格**
- 口语化、轻松自然，去掉"论文感"和"说明书感"
- 开门见山，删掉冗余铺垫
- 长段落拆短，每段不超过 3~4 行
- 结尾加一句互动引导（如"你现在用的是哪个？欢迎评论区交流～"）

**技术内容处理**
- 去除代码块，用日常语言描述其功能
- 将"终端""命令行""CLI"等术语替换为"需要做一些技术操作"或"打开应用就能用"等普通人能理解的表达
- 专业名词第一次出现时可简短解释，之后正常使用

**准确性**
- 保留原文核心事实和观点，不得歪曲或过度简化导致失真
- 如果原文某个说法对普通读者不适用（如"适合程序员"），根据实际情况调整定位，但不能改变事实

### 第三步：格式转换

| 公众号格式 | 小红书格式 |
|---|---|
| `## 标题` | emoji + 粗体标题（如 `## 💻 Code：真正能帮你把事做完的那个`） |
| `- 列表项` | ✅ / ❌ / 👉 等符号开头 |
| 代码块 | 删除，用文字描述替代 |
| 粗体强调 | 保留 `**加粗**` |
| 表格 | 保留，列数过多时拆分 |
| 分隔线 `---` | 保留，用于分节 |

### 第四步：标题处理

**直接使用原文标题**，不做修改。

### 第五步：输出文件

将改写结果输出为 `.md` 文件，文件名格式：`{原文主题}.md`

文件结构：
```
# {原文标题}

{正文内容}
```

---

## 注意事项

- **不要过度娱乐化**：小红书风格轻松，但不等于内容空洞。保持信息密度。
- **不要乱用 emoji**：每个段落开头一个 emoji 做标记即可，正文中不要堆砌。
- **表格优先保留**：小红书支持 Markdown 表格，且对比类内容用表格比列表更清晰。
- **文件必须输出**：每次改写都要生成 `.md` 文件供用户下载，不能只在对话中展示文字。
