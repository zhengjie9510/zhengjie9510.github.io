# 博客全流程工作

覆盖技术博客文章的完整生命周期：写作创作、编辑重组、润色优化、发布前检查。根据用户请求的场景，选择对应的流程执行。

---

## 一、写作/创作

新建文章、起草内容、补充素材。

### front matter 规范

```yaml
---
layout: post
title: "文章标题"
date: YYYY-MM-DD HH:MM:SS +0800
categories: [分类]
tags: [标签1, 标签2]
---
```

### 内容结构建议

1. **开头**：用一两段话交代背景和核心问题，引出核心概念。
2. **定义句**：在前 2-3 段或第一个 `##` 之前，给出核心概念的一句话定义，单独成段并加粗。
3. **正文**：按逻辑分层展开，每个 `##` 节聚焦一个子话题。
4. **结尾**：总结要点，或给出延伸思考方向。

### 图片准备

- 配图使用语义化文件名：`文章 slug-语义.png`，如 `context-engineering-cover.png`
- 存放路径：`assets/img/posts/`
- 宽度不超过 1200px，单张不超过 1MB

## 二、编辑/重组

对已有文章进行结构调整、内容增删、段落重组。

### 结构调整

- [ ] **层级合理**：`##` 节的数量和顺序是否符合逻辑？是否需要合并或拆分？
- [ ] **篇幅均衡**：各节篇幅是否过于悬殊？过长的节是否应拆分，过短的节是否应合并？
- [ ] **过渡自然**：节与节之间是否有承上启下的衔接，还是硬切？

### 内容增删

- [ ] **删冗余**：重复表述、与主题无关的旁枝、过度铺垫是否需要精简？
- [ ] **补缺失**：关键概念是否缺少定义？逻辑链是否有断层？
- [ ] **保风格**：新增内容是否与原文的语气和深度一致？

### 重组原则

- 最小改动：只做必要的结构调整，不重写原文。
- 先读后改：通读全文理解脉络后再动手，避免局部修改破坏整体逻辑。

## 三、润色/优化

可读性细节优化：概念引入、公式解释、段落过渡、术语统一、排版美化等。

### 内容组织

- [ ] **先定义，再展开**：读者应该在进入"为什么需要""如何实现"等技术细节之前，先明白"这到底是什么"。
- [ ] **最小改动**：已有内容只做必要的突出或概括化调整，不新增大段内容。
- [ ] **不提前剧透**：概念定义句应避免列举后文才会详细讲解的具体技术手段，用概括性表达替代。

### 概念引入

- [ ] **是否存在定义**：文章前半部分（前 2-3 段或第一个 `##` 之前）是否有对核心概念的一句话定义？
- [ ] **是否够突出**：定义句是否混在大段文字中、没有视觉焦点？如果是，将其**单独成段并加粗**。
- [ ] **是否太抽象**：定义是否使用了过于笼统的技术词汇（如"信息流"），可以替换为更直观的日常表达（如"内容""信息"）？
- [ ] **是否太具体**：定义中是否提前列举了后文才会展开的具体方法或技术？如果是，将其替换为**概括性表达**（如"一系列做法"）。
- [ ] **是否说清边界**：定义是否让读者明白"它是什么"以及"它不是什么"？必要时用"不是…而是…"的对比句式增强清晰度。

### 公式与变量

- [ ] **变量有解释**：公式中的每个符号（如 $I_t$、$C_t$、$O_t$）是否都有说明？优先用日常词汇（"用户输入""上下文内容""模型输出"）而非英文缩写。
- [ ] **公式有大白话**：如果公式抽象，是否补了一句口语化的解释，说明整个式子在干什么？
- [ ] **避免吓人符号**：如"$t = 1$ 到 $\infty$"对非技术读者不友好，可改为"每一步都重复这个循环"。

### 段落与过渡

- [ ] **段落有过渡**：从一个技术点切换到另一个时，是否有承上启下的句子？避免硬切（如直接出现"通过任务级的隔离……"）。
- [ ] **不说价值判断**：用"还有一种做法"而非"更合理的做法"。

### 词汇与术语

- [ ] **术语统一**：正文中统一用中文表达，不混用英文缩写。
- [ ] **同义词不堆砌**：语义重复时选一个即可。
- [ ] **避免生僻外来借词**：优先使用大众日常熟悉的中文表达，不使用小众圈子或外语借词替代日常词汇。
- [ ] **抽象配例子**：技术名词（如"窗口限制"）对非专业读者太抽象时，是否加了带数字的具体场景？
- [ ] **能用类比就不用黑话**：如用"眼前"和"仓库"替代"模仿计算机内存与硬盘设计"。

### 排版与视觉

- [ ] **层级清晰**：中文数字编号（一、二、三）管层级，表情符号管趣味，不可替代编号。
- [ ] **表情适度**：小节标题或关键标记可加表情符号增加趣味性，不给正文每段都加。
- [ ] **标题风格**：章节标题作为结构导航，避免过度口语化、俚语化或网络梗式表达，优先简洁中性。
- [ ] **重点有标记**：核心痛点、关键结论等可以用表情符号（如 ⚠️、💡）做视觉提示，帮助读者快速定位重点。
- [ ] **引号规范**：中文正文中的引用、比喻、强调使用中文引号（""''）；front matter、HTML 属性、代码块、英文术语保留英文引号。
- [ ] **图片体积合理**：检查文章中引用的本地图片，单张超过 1MB 时应压缩。参考下文"发布前检查"。

## 四、发布前检查

提交前的最终检查：图片处理、格式修正、完整性验证。

### 图片体积

- [ ] **定位图片**：扫描 markdown 中的本地图片引用（`![alt](/assets/img/...)` 或 `<img src="/assets/img/...">`），列出对应文件路径。
- [ ] **检查大小**：`ls -lh` 查看各图片体积，超过 1MB 的标记为待压缩。
- [ ] **压缩后更新引用**：压缩转换格式（如 `.png` → `.jpg`）后，同步更新 markdown 中的文件扩展名。

**压缩命令（macOS `sips`）：**
```bash
sips -Z 800 input.png --out output.png -s format png
```

### 图片命名

- [ ] **语义化文件名**：不要保留 `图片 1.png`、`img.png`、模型默认生成名这类不可维护的名称。
- [ ] **统一格式**：使用"文章名称 slug-语义"格式，不包含文件名前面的日期部分；例如 `harness-engineering-cover.png`、`harness-engineering-three-layers.png`。
- [ ] **语义字段用英文短词**：只保留 2 到 4 个能看懂用途的英文词，使用连字符连接，便于检索和批量处理。
- [ ] **重命名后同步更新正文引用**：修改 Markdown 或 HTML 中的图片路径时，文件名和引用必须同时调整，避免留下失效链接。

### 引号修正

扫描文章中的英文双引号 `"` 和单引号 `'`，判断是否处于中文语境：

- **应替换为中文引号**：正文中的词汇引用、比喻性表达、对话示例。
- **应保留英文引号**：YAML front matter 值、HTML 标签属性、Markdown 链接、代码块、纯英文术语或短语。

使用最小化替换，逐对确认上下文，避免误伤代码或链接。

### 完整性验证

确认文章满足：

- front matter 字段完整（layout、title、date、categories、tags）。
- 读者第一次接触核心概念时，读完定义句就能大致明白它指什么。
- 没有提前暴露后文的具体技术细节。
- 公式和变量对非专业读者友好。
- 段落之间衔接自然，不生硬。
- 层级结构清晰，表情符号点缀得当。
- 中文正文中的引号已统一为中文引号，front matter / HTML / 代码 / 英文术语中的英文引号未被误改。
- 配图文件名已使用语义化命名（"文章 slug-语义"格式），无 `图片 1.png` 等默认名。
- 文章中引用的本地图片单张不超过 1MB（如进行了压缩，路径和扩展名已同步更新）。

## 润色示例

### 示例 A：定义太平淡

**原文：**
```markdown
为了让大语言模型能够处理长程任务，上下文工程成为了不可或缺的技术底层。它是一套由开发者设计的系统性机制，用于管理传递给 LLM 的信息流。在此系统中，AI Agent 扮演着……
```

**问题：** 定义句混在段落中，没有视觉焦点，读者容易滑过去。

**优化后：**
```markdown
为了让大语言模型能够处理长程任务，上下文工程成为了 AI Agent 不可或缺的技术底层。

**所谓上下文工程，就是开发者围绕"该给模型看什么"这个问题，所采用的一系列做法。** 它不是简单地把更多信息塞给模型……
```

### 示例 B：定义太具体

**原文：**
```markdown
……而是通过精密的筛选、编排与压缩，在每一轮推理中让模型看到的都是最相关、最精炼的信息。
```

**问题：** "筛选、编排与压缩"是后文才会展开的具体方法，放在定义里提前剧透，且让定义变得像方法罗列。

**优化后：**
```markdown
……而是通过一系列手段对信息进行处理，在每一轮推理中让模型看到的都是最相关、最精炼的信息。
```

### 示例 C：词汇不直观

**原文：**
```markdown
……专门用于管理传递给大语言模型（LLM）的信息流。
```

**问题：** "信息流"过于抽象和技术化。

**优化后：**
```markdown
……专门用于管理传递给大语言模型（LLM）的内容。
```

### 示例 D：段落过渡生硬

**原文：**
```markdown
### 3. 子智能体

通过任务级的隔离，可以实现 Token 的瞬时清理。
```

**问题：** 从前面的"多层级记忆"直接硬切到"子智能体"，没有过渡。

**优化后：**
```markdown
### 3. 子智能体

除了把信息整理归类，还可以换个思路：把大块任务拆出去独立执行，主 Agent 只看最终结果，过程信息不会涌入主上下文。
```

### 示例 E：公式缺少解释

**原文：**
```markdown
$$
O_t = LLM(I_t, C_t)
$$
```

**问题：** 非专业读者不知道 $I_t$、$C_t$、$O_t$ 是什么。

**优化后：**
```markdown
$$
O_t = LLM(I_t, C_t)
$$

其中，$I_t$ 是每一步的新输入，$C_t$ 是之前累积下来的全部内容，$O_t$ 是模型这一步给出的回答。
```

### 示例 F：表情符号与层级结合

**原文：**
```markdown
## 为什么需要上下文工程？
## 核心公式
## 核心技术手段详解
  ### 压缩与替换
  ### 多层级记忆架构
```

**问题：** 只有表情符号没有编号，层级结构不清晰。

**优化后：**
```markdown
## 一、❓ 为什么需要上下文工程？
## 二、📐 核心公式
## 三、🛠️ 核心技术手段详解
  ### 🗜️ 压缩与替换
  ### 🧠 多层级记忆架构
```

### 示例 G：引号混用

**原文：**
```markdown
上下文工程的核心目标是寻找大语言模型输入的"黄金比例"。LLM 本质上是在做"文字接龙"。
```

**问题：** 中文正文中使用了英文引号，排版不规范。

**优化后：**
```markdown
上下文工程的核心目标是寻找大语言模型输入的"黄金比例"。LLM 本质上是在做"文字接龙"。
```
