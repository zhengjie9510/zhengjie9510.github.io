# 博客文章润色与可读性优化

检查技术博客文章的可读性细节（概念引入、公式解释、段落过渡、术语统一、抽象/具体平衡、排版美化等），并在必要时进行最小化润色。

## 核心原则

1. **先定义，再展开**：读者应该在进入"为什么需要""如何实现"等技术细节之前，先明白"这到底是什么"。
2. **最小改动**：如果文章已有定义，只做必要的突出或概括化调整，不新增大段内容。
3. **不提前剧透**：概念定义句应避免列举后文才会详细讲解的具体技术手段，应使用概括性表达。
4. **数字管层级，表情管趣味**：给文章加表情符号美化排版时，中文数字编号（一、二、三）负责标明层级结构，表情符号负责增加趣味性和视觉区分。两者要分工明确，不能为了表情而丢掉层级清晰度。

## 检查清单

### 概念引入

- [ ] **是否存在定义**：文章前半部分（前 2-3 段或第一个 `##` 之前）是否有对核心概念的一句话定义？
- [ ] **是否够突出**：定义句是否混在大段文字中、没有视觉焦点？如果是，将其**单独成段并加粗**。
- [ ] **是否太抽象**：定义是否使用了过于笼统的技术词汇（如"信息流"），可以替换为更直观的日常表达（如"内容""信息"）？
- [ ] **是否太具体**：定义中是否提前列举了后文才会展开的具体方法或技术？如果是，将其替换为**概括性表达**（如"一系列做法"）。
- [ ] **是否说清边界**：定义是否让读者明白"它是什么"以及"它不是什么"？必要时用"不是…而是…"的对比句式增强清晰度。

### 公式与变量

- [ ] **变量有解释**：公式中的每个符号（如 $I_t$、$C_t$、$O_t$）是否都有说明？优先用日常词汇（"用户输入""上下文内容""模型输出"）而非英文缩写。
- [ ] **公式有大白话**：如果公式抽象，是否补了一句口语化的解释，说明整个式子在干什么？
- [ ] **避免吓人符号**：如"$t = 1$ 到 $\infty$"对非技术读者不友好，可改为"每一步都重复这个循环"。

### 段落与过渡

- [ ] **段落有过渡**：从一个技术点切换到另一个时，是否有承上启下的句子？避免硬切（如直接出现"通过任务级的隔离……"）。
- [ ] **不说价值判断**：不要说"更合理的做法""更彻底的做法"，用"还有一种做法"保持中立。

### 词汇与术语

- [ ] **术语统一**：正文中是否混用了英文缩写和中文（如 Sub-agent / 子智能体、ACE / 自动化上下文工程）？统一用中文。
- [ ] **同义词不堆砌**：如"手段和方法"语义重复，选一个即可。
- [ ] **抽象配例子**：技术名词（如"窗口限制"）对非专业读者太抽象时，是否加了带数字的具体场景？
- [ ] **能用类比就不用黑话**：如用"眼前"和"仓库"替代"模仿计算机内存与硬盘设计"。

### 排版与视觉

- [ ] **层级清晰**：一级标题（`##`）如果有编号（一、二、三），应保留编号以标明层级；表情符号可以放在编号之后或标题文字之前，但不能替代编号。
- [ ] **表情适度**：给小节标题（`###`）或关键标记加表情符号可以增加趣味性，但不要给正文每一段都加，避免视觉噪音。
- [ ] **重点有标记**：核心痛点、关键结论等可以用表情符号（如 ⚠️、💡）做视觉提示，帮助读者快速定位重点。
- [ ] **引号规范**：中文正文中的引用、比喻、强调应使用中文引号（“”‘’），而非英文引号（""''）。需保留 front matter、HTML 属性、代码块、英文术语（如 `"Lost in the Middle"`）中的英文引号。
- [ ] **图片体积合理**：检查文章中引用的本地图片，单张超过 1MB 时应压缩。参考下文"步骤 3"。

### 图片体积（附带检查）

- [ ] **定位图片**：扫描 markdown 中的本地图片引用（`![alt](/assets/img/...)` 或 `<img src="/assets/img/...">`），列出对应文件路径。
- [ ] **检查大小**：`ls -lh` 查看各图片体积，超过 1MB 的标记为待压缩。
- [ ] **压缩后更新引用**：压缩转换格式（如 `.png` → `.jpg`）后，同步更新 markdown 中的文件扩展名。

## 处理流程

### 步骤 1：定位核心概念

从文章标题和 front matter 中提取核心概念（如标题为"一文读懂 Context Engineering"，核心概念就是"上下文工程 / Context Engineering"）。

### 步骤 2：逐项检查

按照"检查清单"依次确认。发现问题时，优先做最小化修改，不破坏原文风格和结构。

### 步骤 3：检查并压缩本地图片（如适用）

扫描文章中的本地图片引用，检查体积并处理：

**1. 列出文章引用的图片：**
```bash
# 根据文章 slug 查找对应图片
cd /Users/zhengjie/Documents/zhengjie9510.github.io
ls -lh assets/img/posts/{slug}-*
```

**2. 压缩标准：**

| 参数 | 值 |
|------|------|
| 宽度 | 800px |
| 输出格式 | PNG |

**3. 压缩命令（macOS `sips`）：**
```bash
sips -Z 800 input.png --out output.png -s format png
```

**4. 更新 markdown 引用：**
压缩后覆盖原文件，保持路径不变。

### 步骤 4：修正引号（如适用）

扫描文章中的英文双引号 `"` 和单引号 `'`，判断是否处于中文语境：

- **应替换为中文引号**：正文中的词汇引用、比喻性表达、对话示例（如 `"断片"`、`"经纪人"`）。
- **应保留英文引号**：YAML front matter 值、HTML 标签属性、Markdown 链接、代码块、纯英文术语或短语。

使用最小化替换，逐对确认上下文，避免误伤代码或链接。

### 步骤 5：验证修改

确认修改后满足：

- 读者第一次接触这个概念时，读完定义句就能大致明白它指什么。
- 没有提前暴露后文的具体技术细节。
- 公式和变量对非专业读者友好。
- 段落之间衔接自然，不生硬。
- 层级结构清晰，表情符号点缀得当。
- 没有新增与原文风格不符的大段内容。
- 中文正文中的引号已统一为中文引号，front matter / HTML / 代码 / 英文术语中的英文引号未被误改。
- 文章中引用的本地图片单张不超过 1MB（如进行了压缩，路径和扩展名已同步更新）。

## 示例

### 示例 A：定义太平淡

**原文：**
```markdown
为了让大语言模型能够处理长程任务，上下文工程成为了不可或缺的技术底层。它是一套由开发者设计的系统性机制，用于管理传递给 LLM 的信息流。在此系统中，AI Agent 扮演着……
```

**问题：** 定义句混在段落中，没有视觉焦点，读者容易滑过去。

**优化后：**
```markdown
为了让大语言模型能够处理长程任务，上下文工程成为了 AI Agent 不可或缺的技术底层。

**所谓上下文工程，就是开发者围绕"该给模型看什么"这个问题，所采用的一系列做法。** 它不是简单地把更多信息塞给模型……
```

### 示例 B：定义太具体

**原文：**
```markdown
……而是通过精密的筛选、编排与压缩，在每一轮推理中让模型看到的都是最相关、最精炼的信息。
```

**问题：** "筛选、编排与压缩"是后文才会展开的具体方法，放在定义里提前剧透，且让定义变得像方法罗列。

**优化后：**
```markdown
……而是通过一系列手段对信息进行处理，在每一轮推理中让模型看到的都是最相关、最精炼的信息。
```

### 示例 C：词汇不直观

**原文：**
```markdown
……专门用于管理传递给大语言模型（LLM）的信息流。
```

**问题：** "信息流"过于抽象和技术化。

**优化后：**
```markdown
……专门用于管理传递给大语言模型（LLM）的内容。
```

### 示例 D：段落过渡生硬

**原文：**
```markdown
### 3. 子智能体

通过任务级的隔离，可以实现 Token 的瞬时清理。
```

**问题：** 从前面的"多层级记忆"直接硬切到"子智能体"，没有过渡。

**优化后：**
```markdown
### 3. 子智能体

除了把信息整理归类，还可以换个思路：把大块任务拆出去独立执行，主 Agent 只看最终结果，过程信息不会涌入主上下文。
```

### 示例 E：公式缺少解释

**原文：**
```markdown
$$
O_t = LLM(I_t, C_t)
$$
```

**问题：** 非专业读者不知道 $I_t$、$C_t$、$O_t$ 是什么。

**优化后：**
```markdown
$$
O_t = LLM(I_t, C_t)
$$

其中，$I_t$ 是每一步的新输入，$C_t$ 是之前累积下来的全部内容，$O_t$ 是模型这一步给出的回答。
```

### 示例 F：表情符号与层级结合

**原文：**
```markdown
## 为什么需要上下文工程？
## 核心公式
## 核心技术手段详解
  ### 压缩与替换
  ### 多层级记忆架构
```

**问题：** 只有表情符号没有编号，层级结构不清晰。

**优化后：**
```markdown
## 一、❓ 为什么需要上下文工程？
## 二、📐 核心公式
## 三、🛠️ 核心技术手段详解
  ### 🗜️ 压缩与替换
  ### 🧠 多层级记忆架构
```

**原则：** 一级标题保留中文数字（一、二、三）管层级，表情符号放在数字之后增加趣味；二级小节可以只用表情符号，因为读者已经通过一级标题感知到了层级。

### 示例 G：引号混用

**原文：**
```markdown
上下文工程的核心目标是寻找大语言模型输入的"黄金比例"。LLM 本质上是在做"文字接龙"。
```

**问题：** 中文正文中使用了英文引号，排版不规范。

**优化后：**
```markdown
上下文工程的核心目标是寻找大语言模型输入的“黄金比例”。LLM 本质上是在做“文字接龙”。
```

**原则：** 中文语境下的引用、比喻、强调统一使用中文引号；front matter、HTML 属性、代码、英文术语保留英文引号。
