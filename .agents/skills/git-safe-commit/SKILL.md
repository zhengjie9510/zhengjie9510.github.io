# Git 安全提交 Skill

## 执行原则

### ✅ 自动执行（无需确认）
以下操作可以直接执行，不需要询问用户：

- **文件修改**：编辑、格式化、规范化文件内容
- **文件重命名**：按规范重命名文件
- **添加文件到 Git**：`git add -A` 或添加特定文件
- **查看状态**：`git status`、`git diff` 等查看类命令
- **下载/复制资源**：下载图片、复制文件等
- **删除临时文件**：删除 .DS_Store 等系统文件
- **创建目录**：创建所需的文件夹结构

### ⚠️ 必须询问确认
以下操作在执行前**必须**询问用户并获得明确同意：

- **Git Commit**：提交更改到本地仓库
- **Git Push**：推送到远程仓库

## 询问方式

使用 **AskUserQuestion 工具** 提供选项按钮，让用户点击选择，而不是输入文字。

### Commit 前询问
选项：
- ✅ **是，提交** - 执行 git commit，使用建议的提交信息
- ❌ **否，先不提交** - 保留更改在工作区，稍后手动提交  
- ✏️ **修改提交信息** - 让用户输入自定义提交信息

### Push 前询问
选项：
- ✅ **是，推送** - 执行 git push origin main
- ❌ **否，先不推送** - 保留提交在本地，稍后手动推送

## 完整工作流程示例

**场景：规范化一篇新博客文章**

```
用户：帮我处理这篇新博客

AI：
1. [自动] 读取原文件
2. [自动] 重命名为规范文件名
3. [自动] 添加 front matter
4. [自动] 插入公众号引流
5. [自动] 下载封面图片
6. [自动] git add -A
7. [询问] "更改完成，是否提交？提交信息：'feat: 新增XXX文章'"
   → 用户确认
8. [自动] git commit
9. [询问] "是否推送到远程？"
   → 用户确认
10. [自动] git push
```

## 特殊情况处理

### 如果用户说"先不提交"
- 保留所有已完成的文件修改
- 告知用户："更改已保存在工作区，稍后可以通过 'git add -A && git commit' 手动提交"

### 如果用户说"提交但不要push"
- 执行 commit
- 询问："已提交到本地，是否推送？"
- 如用户说否，告知："稍后可通过 'git push origin main' 推送"

### 批量处理多篇文章时
- 可以一次性处理所有文件修改
- 但最后统一询问一次 commit 和 push
- 示例："已完成 3 篇文章的规范化，是否统一提交？"

## 记忆点

- ✅ 可以自动做：改文件、加文件、看状态、下图片、删临时文件
- ⚠️ 必须问：commit 和 push
- 默认假设：用户信任文件修改，但需要掌控代码仓库的提交历史
